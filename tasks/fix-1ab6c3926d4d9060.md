# fix-1ab6c3926d4d9060 — E6T1/E6T2: Read-then-write race condition on preference/query updates

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 SATB

`setPrefs` (E6T1.ts:38-39) reads existing prefs, merges, then writes — a non-atomic get-then-set. `/query` (E6T2.ts:59-61) does the same: read current queries, prepend, then write. Under concurrent access (multiple bot instances against shared Redis, or async interleaving), these can silently lose writes because two readers can see the same stale value and each overwrite the other's update.
