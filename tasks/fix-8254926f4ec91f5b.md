# fix-8254926f4ec91f5b — Extension mismatch: cleanup can miss files, causing storage leaks

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 SATB

E7T1 stores image files using the actual file extension from Telegram's `file_path` (e.g. `.webp`, `.bmp`, `.tiff`), but **always** maps the `mimeType` metadata to `"image/jpeg"` for any extension other than `png` or `gif` (`src/handlers/E7T1.ts:125`).

E7T2's `/cleanupcache` derives the file extension from the stored `mimeType` to locate and delete the file on disk (`src/handlers/E7T2.ts:9-12`). For any non-jpg/jpeg/png/gif image, this produces an extension (`"jpeg"`) that does not match the actual file on disk. The fallback `if (ext === "jpeg") deleteFile(meta.id, "jpg")` (`src/handlers/E7T2.ts:44`) only saves the `.jpg` case.

**Consequence:** images stored with extensions like `.webp`, `.bmp`, `.tiff` etc. are orphaned on cleanup — the metadata is deleted but the file remains on disk forever, causing unbounded storage growth and a growing discrepancy between `/imagestatus` file counts and metadata counts.

**Fix:** either store the actual file extension (or the original `filePath`) in `ImageMetadata` so E7T2 can reconstruct the correct filename, or derive `mimeType` from the actual extension instead of defaulting to `"image/jpeg"`.
